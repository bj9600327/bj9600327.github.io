jekyll update

运营文档

撰写前

  1. 检查当前所在特性分支是否有未提交的工作。如果有， stash 或 commit
  2. 切换到与待撰写内容匹配的特性分支
    1. 特性分支命名规范：用-分隔的纯小写单词
    2. 确保所有特性分支两两之间没有共同追踪同一个文件，以便章鱼合并
    3. 对容易混淆的分支名，用带附注的标签描述。该标签指向首次分离于master的提交。标签名为：descript_BRANCH

撰写时

  1. _posts 文件夹的规范
    1. 命名规范。一旦提交，原则上不能重命名！
      1. 普通文章：yyyy-mm-dd-INDEX.markdown
      2. csv表格：yyyy-mm-dd-spilt-by-sub.markdown
    2. 内容规范
      1. 正文排版：以《了不起的 Markdown 》推荐格式为准
      2. 引用视频规范：发布到b站后生成博客链接引用
      3. 前六行格式：与 yyyy-mm-dd-welcome-to-jekyll.markdown 一致
        1. 标题规范：尽量用中文
        2. 分类规范：与分支名一致或更细分
        3. 时间规范：
          1. 撰写时间20yy-mm-dd hh:mm:ss +0800
          2. 置顶格式40yy-mm-dd hh:mm:ss +0800
          3. 置底格式00yy-mm-dd hh:mm:ss +0800
  2. image 文件夹的规范
    1. 命名规范：INDEX.主码.图片原生后缀
      1. 即使有两个 markdown 文件引用同一张图片，也要分两个不同名的图片副本
      2. 一旦提交，禁止重命名！禁止编辑与更换清晰度！

撰写后

  1. 提交描述的规范：把 git 自动生成的“要提交的变更”列表取消注释
    1. 如果仅修改 markdown 文件的正文，需描述标题:修改的概要
    2. 如果仅修改 markdown 文件的 title 字段，需描述title:旧=>新
    3. 如果仅修改 markdown 文件的 date 字段，需描述date:旧=>新
    4. 如果仅修改 markdown 文件的 categories 字段，需描述categories:旧=>新
    5. 如果仅删除文件，需描述trash:文件名
    6. 如果要做上述两种及以上的修改，请分割成多个提交
  2. 检查git stash show，如果有工作未完成，立即做或者做好计划。
  3. 如果新笔记不需要在电脑关闭时访问，暂缓发布，把特性分支 push 到远端用于备份。

发布

  1. 用git branch --no-merged master查找所有未合并入 master 的特性分支。
  2. 用章鱼合并策略和-no-ff选项，把除独立发布分支以外的所有特性分支一次性合并到 master
  3. 更新部署页面
  4. 最后在移动端测试页面。
